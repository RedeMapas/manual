> Uma plataforma livre de mapeamento e gestão cultural usada por municípios, estados e o governo federal brasileiro.

Mapas Culturais is an open-source platform for cultural mapping and management. Used by hundreds of Brazilian municipalities, state governments, and federal institutions, it brings together cultural agents, spaces, events, and projects in a single collaborative environment — supporting the complete cultural funding journey from registration to final reporting.

The platform is maintained by a community network of developers and public managers, and is pursuing recognition as **Software Público** by the Brazilian government and as a **Digital Public Good** by the United Nations.

## What Mapas Culturais does

Mapas Culturais provides a web environment for mapping, publishing, and managing cultural assets. Its core entities are:

* **Agents** — individuals, collectives, and organizations with geolocation and profile data
* **Spaces** — theaters, museums, cultural centers, and alternative venues
* **Events** — scheduled activities with recurrence and geographic reference
* **Projects** — grouped cultural initiatives with phases and participants
* **Opportunities / Editais** — open calls with multi-phase evaluation workflows
* **Registrations** — applications tied to opportunities, tracked through the full evaluation lifecycle

Together these entities implement the **Jornada Completa do Fomento Cultural** (complete cultural funding journey): from an agent's first registration on the platform through submitting to an edital, evaluation, selection, monitoring, and final accounting.

## Architecture overview

Mapas Culturais is a server-side PHP application built around Slim, Doctrine, PostgreSQL/PostGIS, and a modular theme and module system. In practice, the codebase is split between framework core, feature modules, installation themes, and deployment/configuration layers.

For the full stack, directory layout, runtime flows, and core abstractions, use the canonical architecture guide: [Architecture](/mnt/data/code/redemapas/manual/content/docs/devs/development/architecture.mdx).

## Branch structure

The RedeMapas repository maintains two parallel development lines:

| Branch         | Synchronized with               | Purpose                                               |
| -------------- | ------------------------------- | ----------------------------------------------------- |
| `main`         | —                               | RedeMapas production branch                           |
| `develop`      | `mapasculturais/mapasculturais` | Community line — integrates open-source contributions |
| `develop-minc` | `culturagovbr/mapasculturais`   | MinC line — integrates Ministério da Cultura features |

When contributing, identify which line your change belongs to and open a pull request against `develop` or `develop-minc`. Use the `-minc` suffix on branch names (e.g., `feat/something-minc`) for work on the MinC line.

## RedeMapas in practice

The reference installation documented here is not a generic "vanilla Mapas" setup. It is the **RedeMapas / Rede das Artes** line, which combines:

* the Mapas Culturais core
* a dedicated `RedeMapas` theme under `src/themes/`
* deployment conventions for Docker Compose and Kubernetes
* production concerns such as subsites, Web Push, and MinC-compatible authentication flows

That distinction matters when you read or change the codebase. Some behavior lives in the framework core, some in modules, and some only in the active installation theme.

## Contribution model

When starting a change, answer these questions first:

1. Is this a **framework/core** change that belongs upstream?
2. Is this a **RedeMapas installation** concern that should stay in the local theme or config?
3. Is this specific to the **community line** (`develop`) or the **MinC line** (`develop-minc`)?

As a rule of thumb:

* framework and reusable platform improvements should target the appropriate upstream line
* installation-specific UI and integration behavior should stay in the RedeMapas theme or deployment config
* branch naming should make the target explicit, especially for MinC work

## Active installations

Mapas Culturais is deployed across dozens of Brazilian and international installations:

**Federal / International**

* [SNIIC](https://mapa.cultura.gov.br/) — Sistema Nacional de Informações e Indicadores Culturais
* [Cultura Viva](https://culturaviva.cultura.gov.br/) — federal program for cultural points
* [Rede das Artes](https://rededasartes.cultura.gov.br/) — Ministério da Cultura platform
* [IberculturaViva](https://mapa.iberculturaviva.org/) — Ibero-American community culture network
* [Mapa Uruguai](https://culturaenlinea.uy/) — Uruguay's national cultural map

**State installations (selection)**

Amapá, Ceará, Espírito Santo, Goiás, Maranhão, Mato Grosso, Pará, Pernambuco, Paraíba, Piauí, Tocantins, and more than 20 others.
