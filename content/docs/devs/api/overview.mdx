> Introdução à API REST do Mapas Culturais: entidades, endpoints, métodos HTTP, autenticação e formato de resposta.

## Introdução

O Mapas Culturais expõe uma API REST que permite a sistemas externos ler, criar, atualizar e remover entidades culturais. Todas as respostas são em JSON. A API está disponível em qualquer instalação e segue uma estrutura de URL consistente entre os tipos de entidade.

## Base URL

```
{baseUrl}/api/{entity}/{action}
```

| Segment   | Descrição                                                       |
| --------- | ----------------------------------------------------------------- |
| `baseUrl` | URL raiz da instalação, por exemplo `https://mapas.cultura.gov.br` |
| `entity`  | Um dos tipos de entidade suportados (veja abaixo)                  |
| `action`  | A operação a executar, como `find`, `findOne`, `describe` etc.     |

**Exemplo:**

```bash  theme={null}
# Busca todos os agentes publicados, retornando apenas id e nome
curl "https://your-instance.example.com/api/agent/find?@select=id,name&status=EQ(1)"
```

## Entidades suportadas

| Slug da entidade | Descrição                                                |
| --------------- | -------------------------------------------------------- |
| `agent`         | Indivíduos ou coletivos cadastrados na plataforma        |
| `space`         | Espaços culturais físicos ou virtuais                    |
| `event`         | Eventos culturais com ocorrências e agenda               |
| `project`       | Projetos culturais que agrupam eventos ou oportunidades  |
| `opportunity`   | Editais, chamadas e processos de inscrição               |
| `registration`  | Inscrições enviadas por agentes em oportunidades         |

## Métodos HTTP e ações

| Método HTTP | Padrão de URL            | Descrição                                     |
| ----------- | ------------------------ | --------------------------------------------- |
| `GET`       | `/api/{entity}/find`     | Retorna um array de entidades compatíveis     |
| `GET`       | `/api/{entity}/findOne`  | Retorna uma única entidade compatível         |
| `GET`       | `/api/{entity}/describe` | Retorna o schema da entidade com metadados    |
| `POST`      | `/{entity}/index`        | Cria uma nova entidade                        |
| `PATCH`     | `/{entity}/single/{id}`  | Atualiza parcialmente uma entidade existente  |
| `PUT`       | `/{entity}/single/{id}`  | Substitui integralmente uma entidade existente |
| `DELETE`    | `/{entity}/single/{id}`  | Faz exclusão lógica de uma entidade           |

<Note>
  Operações de escrita usam URLs **sem** o prefixo `/api/`. Exemplo: `POST /agent/index` para criar um agente e `PATCH /agent/single/42` para atualizar o agente 42.
</Note>

<Note>
  Operações de escrita (`POST`, `PATCH`, `PUT`, `DELETE`) exigem autenticação via JWT. Requisições somente leitura com `find` e `findOne` podem ser feitas sem credenciais quando a entidade é pública.
</Note>

## Formato de resposta

### `find` — array de objetos

```json  theme={null}
[
  {
    "id": 42,
    "name": "Companhia do Teatro",
    "status": 1,
    "createTimestamp": { "date": "2023-04-01 10:00:00.000000", "timezone_type": 3, "timezone": "America/Sao_Paulo" },
    "updateTimestamp": { "date": "2024-01-15 14:30:00.000000", "timezone_type": 3, "timezone": "America/Sao_Paulo" },
    "@entityTipo": "agent"
  }
]
```

### `findOne` — objeto único

```json  theme={null}
{
  "id": 42,
  "name": "Companhia do Teatro",
  "status": 1,
  "createTimestamp": { "date": "2023-04-01 10:00:00.000000", "timezone_type": 3, "timezone": "America/Sao_Paulo" },
  "updateTimestamp": { "date": "2024-01-15 14:30:00.000000", "timezone_type": 3, "timezone": "America/Sao_Paulo" },
  "@entityTipo": "agent"
}
```

### Cabeçalho de metadados de paginação

Quando `@page`, `@offset` ou `@limit` são usados, o servidor adiciona um cabeçalho `API-Metadata` à resposta:

```json  theme={null}
{
  "count": 240,
  "page": 2,
  "limit": 20,
  "numPages": 12,
  "keyword": "",
  "order": "name ASC"
}
```

### Campos comuns da resposta

| Campo             | Tipo    | Descrição                                                                    |
| ----------------- | ------- | ---------------------------------------------------------------------------- |
| `id`              | integer | Identificador único da entidade                                              |
| `name`            | string  | Nome de exibição da entidade                                                 |
| `status`          | integer | `1` = publicado, `0` = rascunho, `-9` = desabilitado, `-10` = lixeira, `-2` = arquivado |
| `createTimestamp` | object  | Data e hora de criação (formato PHP DateTime)                                |
| `updateTimestamp` | object  | Data e hora da última atualização (formato PHP DateTime)                     |
| `@entityTipo`     | string  | Slug do controller que identifica o tipo de entidade                         |

## Códigos de status HTTP

| Código | Significado                                                           |
| ------ | --------------------------------------------------------------------- |
| 200    | Sucesso: a requisição foi processada e um corpo de resposta foi retornado |
| 401    | Não autorizado: autenticação é obrigatória                            |
| 403    | Proibido: o usuário autenticado não tem permissão para essa ação      |
| 404    | Não encontrado: a entidade ou endpoint solicitado não existe          |

## Autenticação

A API suporta dois modos de autenticação:

### Baseada em sessão (navegador)

Quando a API é acessada por um navegador em que o usuário já está autenticado, o cookie de sessão existente é usado automaticamente. Nenhum cabeçalho adicional é necessário.

### JWT / chave de API (servidor para servidor)

Operações de escrita e acesso a dados privados exigem um JSON Web Token (JWT) gerado a partir de um par de chaves de aplicativo registrado. O JWT é validado pelo módulo `Apps` usando a biblioteca `firebase/php-jwt`.

Toda requisição autenticada de escrita deve incluir dois cabeçalhos:

```http  theme={null}
authorization: <your-jwt-token>
MapasSDK-REQUEST: true
```

O cabeçalho `authorization` carrega o JWT. O cabeçalho `MapasSDK-REQUEST: true` sinaliza ao servidor que ele deve retornar JSON em vez de um redirecionamento HTTP.

O payload do JWT deve incluir o campo `pk` com o identificador da chave pública do aplicativo e um campo `tm` com o timestamp Unix atual:

```json  theme={null}
{ "tm": "1493823078.9774", "pk": "your-app-public-key" }
```

O algoritmo de assinatura padrão é `HS256`. A chave privada do aplicativo, registrada na plataforma sob sua conta de usuário, é usada para assinar e verificar o token.

## CORS

CORS support is configurable per installation. Check with the platform administrator for the allowed origins on a specific instance.

## Rate limiting

Rate limiting depends on the installation configuration. There is no built-in global rate limit in the application; infrastructure-level limits (e.g. nginx, CDN) may apply.

## Seleção de campos com `@select`

Todas as requisições `find` e `findOne` aceitam o parâmetro `@select` para controlar quais campos são retornados. Se `@select` for omitido, todos os campos disponíveis serão retornados.

```bash  theme={null}
# Retorna apenas id e nome
curl "https://your-instance.example.com/api/agent/find?@select=id,name"

# Inclui um campo de entidade relacionada
curl "https://your-instance.example.com/api/space/find?@select=id,name,owner.name"
```

See [Consultando a API](/docs/devs/api/querying) for the full query syntax including filters, pagination, and ordering.
